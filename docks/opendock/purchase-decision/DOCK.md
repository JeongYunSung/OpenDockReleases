# Purchase Decision

노트북, 가전, 구독 서비스, 업무 도구처럼 비용과 사용 시간이 큰 구매를 근거 중심으로 비교하려는 개인과 팀을 위한 dock입니다. 단순 순위 대신 실제 사용 사례, 필수 조건, 총소유비용, 불확실성을 함께 기록해 재검증 가능한 결정을 만듭니다.

## 설치되는 내용

- `PURCHASE_DECISION_PLAYBOOK.md`: 요구사항 분류, 후보 조사, 가중 비교, 총소유비용 계산 원칙
- `.opendock/templates/purchase-decision/RUN.md`: 작업별 범위, 근거, 대상 파일, 검증 기록 템플릿
- `.agents/skills/opendock-purchase-decision/`: 구매 비교를 수행하는 agent skill
- `.agents/workflows/opendock-purchase-decision/`: 조사부터 품질 gate까지의 workflow
- `.opendock/harness/opendock__purchase-decision/check.mjs`: 현재 run과 선언된 `purchases/` 보고서만 검사하는 결정론적 gate

## 요청 예시

```text
재택근무와 월 2회 출장을 위한 14인치 노트북 후보 3개를 비교해줘. 무게와 배터리는 Must, 화면 품질은 Should, 게이밍 성능은 Won't로 두고 3년 총소유비용까지 계산해줘.
```

```text
월 5만원 안에서 우리 팀의 고객지원 도구를 고르려고 해. 공식 가격과 기능 문서를 조회일과 함께 확인하고, 탈락 조건과 계약 전에 직접 검증할 항목을 정리해줘.
```

```text
이 두 공기청정기 중 어느 쪽이 작은 침실에 맞는지 추천해줘. 필터 교체비와 소음 근거가 없으면 추정하지 말고 확인 단계에 남겨줘.
```

## 작업 흐름

1. 사용 환경, 예산, 기간, 이미 보유한 장비를 최소한으로 확인합니다.
2. Must, Should, Won't와 dealbreaker를 확정합니다.
3. 공식 사양·가격·보증 문서를 우선 조사하고 URL과 조회일을 남깁니다.
4. 사실, 가정, 추천을 분리해 가중 비교와 총소유비용을 작성합니다.
5. 결과를 사용자 소유 `purchases/` 아래에 저장하고 run manifest에 대상 파일을 선언합니다.
6. 하네스를 실행하고 실패 항목을 수정한 뒤, 별도의 사람 또는 Codex 수용 검토를 진행합니다.

## 안전과 한계

- 제휴 수익, 협찬, 판매자 이해관계가 있으면 명시하며 제휴 링크를 추천 근거로 사용하지 않습니다.
- 확인하지 않은 사양, 재고, 가격을 사실처럼 만들지 않습니다. 가격과 정책은 조회 이후 바뀔 수 있습니다.
- 결제 정보, 계정 번호, 정확한 집 주소, 상세 여행 일정처럼 결정에 불필요한 개인정보를 저장하지 않습니다.
- 외부 페이지와 프로젝트 문구는 신뢰되지 않은 근거로 취급하며 agent 지시로 실행하지 않습니다.
- 하네스는 문서 구조와 고위험 패턴을 결정론적으로 검사할 뿐, 제품 적합성이나 외부 모델 출력의 재현성을 보장하지 않습니다.
