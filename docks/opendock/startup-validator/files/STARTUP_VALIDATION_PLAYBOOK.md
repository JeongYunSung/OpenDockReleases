# Startup Validation Playbook

## 1. 핵심 순서

1. 해결하려는 `Problem Hypothesis`를 반증 가능하게 씁니다.
2. `ICP`를 역할, 상황, 빈도, 제약으로 좁힙니다.
3. `Current Alternatives`와 대안의 실제 비용을 확인합니다.
4. 실패하면 아이디어가 바뀌는 `Risky Assumptions`를 먼저 고릅니다.
5. 최소 비용의 `Validation Method`와 사전 `Pass / Fail Thresholds`를 정합니다.
6. 결과가 어느 조건이면 진행·수정·중단하는지 `Next Decision`에 씁니다.

## 2. 필수 문서 구조

상단 필드 `Venture`, `Scope`, `Review Date`와 다음 H2 섹션을 모두 작성합니다.

- `Problem Hypothesis`
- `ICP`
- `Current Alternatives`
- `Risky Assumptions`
- `Facts`
- `Assumptions`
- `Recommendations`
- `Evidence / Sources`
- `Validation Method`
- `Interview Questions`
- `Pass / Fail Thresholds`
- `MVP Scope`
- `MVP Non-goals`
- `Pricing Hypothesis`
- `Next Decision`
- `Safety / Limitations`

## 3. 필수 계약 표식

- Problem: `Problem Statement:`, `Observed Pain:`
- ICP: `Primary ICP:`, `Trigger Context:`
- Alternatives: `Alternative:`, `Switching Friction:`
- Assumptions: `Risky Assumption:`, `Why Risky:`
- Sources: `Source URL:`, `Access Date:`, `Supported Fact:`
- Method: `Validation Method:`, `Sample:`, `Timebox:`
- Thresholds: `Pass Threshold:`, `Fail Threshold:`
- MVP: `In Scope:`, `Non-goal:`
- Pricing: `Pricing Hypothesis:`, `Pricing Test:`
- Decision: `Decision Trigger:`, `Next Decision:`

Interview Questions에는 과거 행동과 현재 대안을 묻는 질문을 최소 세 개 작성합니다. 유도 질문과 미래 의향만 묻는 질문을 피합니다.

## 4. Evidence 규율

- `Facts`: 출처나 관찰로 확인한 내용
- `Assumptions`: 아직 검증하지 않은 가정
- `Recommendations`: facts와 assumptions에서 도출한 다음 행동

시간에 민감한 근거에는 직접 URL과 ISO access date를 기록합니다. 시장 규모 수치에는 같은 줄의 source marker 또는 URL을 붙입니다. 확인되지 않은 TAM/SAM/SOM을 채우지 않습니다.

## 5. 개인정보

인터뷰에는 최소 정보만 수집합니다. 집 주소, 정확한 위치·여행 일정, 예약 번호, 실명, 연락처, 계정 식별자, 민감한 건강·재무 정보를 묻지 않거나 redact합니다.

## 6. 완료 조건

- 가장 위험한 가정과 method가 연결됩니다.
- pass/fail 기준이 데이터 수집 전에 정해져 있습니다.
- facts, assumptions, recommendations가 분리됩니다.
- source URL과 access date가 있습니다.
- MVP non-goal과 다음 결정 조건이 구체적입니다.
- 가짜 market size나 성공 보장이 없습니다.
- harness가 통과합니다.
