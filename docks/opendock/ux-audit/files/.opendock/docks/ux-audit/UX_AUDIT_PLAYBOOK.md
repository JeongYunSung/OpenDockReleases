# UX Audit Playbook

평소에는 별도 준비 없이 바로 감사를 수행합니다. 작업 메모가 필요하면 선택 템플릿에서 관련 섹션만 사용하고, 사용자가 검토를 요청하면 AI가 현재 결과물만 이 플레이북 기준으로 직접 검토합니다.

## 1. 범위 고정

사용자 유형, 시작·종료 상태, 플랫폼, viewport, 인증 상태, 제외 범위를 먼저 정합니다. 한 run에서 서로 다른 제품 흐름을 억지로 합치지 않습니다.

## 2. 증거 수집

증거는 직접 재현, 화면·DOM 관찰, 제공된 인터뷰·테스트 기록, 접근성 검사 결과처럼 출처가 식별되는 항목만 사용합니다. 각 finding에는 재현 조건과 관찰 위치를 기록합니다. 제공되지 않은 전환율, 이탈률, 사용자 수, 만족도는 만들지 않습니다.

개인 데이터는 목적에 필요한 최소 범위만 사용합니다. 여행·이동 분석에서는 정확한 일정과 예약번호를, 주거·홈 화면에서는 주소와 출입 정보·정확한 위치를, 개인 계정 화면에서는 이름·연락처·계정 식별자를 제거하거나 범주형 값으로 바꿉니다.

## 3. Finding 구조

각 finding은 다음 연결을 갖습니다.

- 관찰: 사용자가 마주치는 구체적 상태
- Evidence: 화면 위치, 재현 단계, 제공 자료 식별자
- Severity: 영향의 크기와 실패 가능성
- Recommendation: 변경 대상과 기대 동작이 분명한 조치
- Priority: 영향, 빈도 근거, 구현 의존성을 함께 고려한 순서

`Severity`는 사용자 피해를, `Priority`는 실행 순서를 뜻합니다. 둘을 같은 값으로 자동 복사하지 않습니다.

## 4. 커버리지 선택

감사 범위와 사용자 흐름에 관련된 항목만 선택합니다. 아래 항목을 모두 별도 섹션으로 작성할 필요는 없습니다.

- Accessibility: keyboard, focus, 이름·역할·값, 대비, 확대, 오류 인지
- Responsive: 작은 viewport, 긴 텍스트, 확대, 방향 전환, overflow
- Copy: 명확한 label, 오류 복구, 위험 행동 확인, 일관된 용어

중요하지만 확인할 수 없는 항목은 적용 범위와 한계로 남깁니다.

## 5. 완료 조건

보고서가 필요하면 `audits/ux/` 아래에 저장합니다. 실제 device, assistive technology, 사용자 테스트가 필요한 항목은 한계로 남깁니다. 검토 요청에서는 AI가 현재 결과의 증거, 심각도, 권고와 한계를 확인하며 경험의 효과를 보장하지 않습니다.
